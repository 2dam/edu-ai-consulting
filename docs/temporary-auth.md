# 임시 인증 (X-User-Id 헤더)

## 현재 방식

이 리포지토리에는 원래 인증이 전혀 구현되어 있지 않았습니다. 커뮤니티 모듈을 위해
최소한의 임시 사용자 식별 방식을 추가했습니다:

1. `POST /community/users`에 닉네임(선택적으로 지역)만 보내면 `User` 행이 생성되고
   `id`가 발급됩니다. 비밀번호도, 세션도, 이메일 인증도 없습니다.
2. 프론트엔드(`dashboard-community/`)는 발급받은 `id`를 `localStorage`에 저장하고, 이후 모든
   쓰기 요청(게시글/댓글 작성, 투표, 신고 등)에 `X-User-Id: <id>` 헤더로 실어 보냅니다.
3. 백엔드(`api/app/auth.py`의 `get_current_user`)는 이 헤더 값을 그대로 신뢰해서
   `User` 행을 조회합니다. 서명도, 만료도, 검증도 없습니다.

## 알려진 한계 (스푸핑 가능)

- `X-User-Id` 헤더는 누구나 임의의 값으로 위조할 수 있습니다 — 다른 사용자인 척
  글을 쓰거나 투표할 수 있습니다.
- 세션 만료가 없어 한 번 발급받은 id는 영구적으로 유효합니다.
- 관리자 권한(`ADMIN_USER_IDS`, 현재 `{1}`로 하드코딩)도 같은 방식으로 취약합니다 —
  `X-User-Id: 1`만 보내면 누구나 관리자 라우트를 호출할 수 있습니다.
- 브라우저를 새로고침하면 `localStorage`의 id 자체는 남아있어 쓰기 요청은 계속
  동작하지만, 프론트엔드의 "로그인 표시(닉네임)"는 사라집니다 (사용자 프로필을
  다시 불러오는 `GET /community/users/{id}` 같은 엔드포인트가 아직 없기 때문).
